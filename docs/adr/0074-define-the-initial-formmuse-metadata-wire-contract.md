# Define the initial FormMuse metadata wire contract

FormMuse V1 will use `page` as the only initial layout value because Hanging Gifts is the only real implementation-ready composition available to establish that vocabulary. The `animation` field will be a non-empty array of unique values drawn from the initial closed set `css`, `motion`, and `gsap`, reflecting the distinct animation technologies materially used by Hanging Gifts. `@gsap/react`, ScrollTrigger, and similar adapters or plugins remain implementation details rather than separate animation-technology values. Additional layout or animation values require evidence from another real template rather than speculative preallocation.

Every `meta.formmuse` object will use the exact strict keys `layout`, `animation`, `appearance`, `fields`, `tags`, `featured`, `status`, `version`, `updated`, `props`, `examples`, `usageNotes`, and `accessibilityNotes`, plus lifecycle-specific `deprecation` data when required. `props` will be a non-empty array of strict `{ name, type, required, description, default? }` records; an optional `default` must be a Transport Value. `examples` will be a non-empty array of strict `{ title, description, path }` records whose paths are safe repository-relative `.tsx` references. `usageNotes` and `accessibilityNotes` will be non-empty arrays of concise non-empty strings. Prop names and example paths will be unique within their arrays.

Lifecycle validation will use a discriminated contract. Draft records and deprecated records must set `featured` to `false`. Published records may set `featured` to either Boolean value. Draft and published records must not contain `deprecation`. Deprecated records must contain the strict object `{ warning, replacement?, installable }`, where `warning` is non-empty, `replacement` is an optional Canonical Slug, and `installable` explicitly records whether the deprecated source remains safe, legally redistributable, and functional. Published installability and draft non-installability remain derived from Publication Status and are not duplicated as metadata flags.

Template Versions in V1 will accept only stable `major.minor.patch` semantic versions with no leading zeroes except the number zero itself. Prerelease and build-metadata forms are rejected because FormMuse V1 has no prerelease channel. Updated dates will remain strict, real ISO calendar dates in `YYYY-MM-DD` form.

`FormMuseMetaSchema` remains the sole runtime and inferred TypeScript definition for FormMuse Metadata. The schema module will also expose a strict registry-boundary contract containing the standard top-level one-item `categories` tuple and `meta.formmuse`; that boundary rejects a Primary Category duplicated in `tags`. It does not redefine the complete shadcn Registry Record. The root registry and deterministic generator will compose the shadcn contract with this boundary in Stage 1.4.
