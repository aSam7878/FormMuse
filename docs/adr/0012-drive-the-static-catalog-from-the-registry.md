# Drive the static catalog from the registry

`registry.json` will be the single source of truth for Form Template installation data, documentation, Template Pages, search, and filters. Standard shadcn registry fields—including `name`, `title`, `description`, `categories`, `files`, `dependencies`, and `registryDependencies`—will remain at the top level of each Registry Record and will not be duplicated.

All custom FormMuse catalog and documentation metadata will live under `meta.formmuse`. This namespace will contain FormMuse-specific values such as layout, animation technology, appearance, included fields, tags, featured status, publication status, Template Version, and updated date. FormMuse will generate client-side search and filtering from these records and will not add a database, ratings, favourites, accounts, or popularity tracking in V1, avoiding metadata drift and runtime catalog infrastructure.

`FormMuseMetaSchema`, implemented with Zod, will be the sole runtime and TypeScript definition for `meta.formmuse`; consumers will use its inferred type rather than a separate handwritten interface. The schema will be strict, use closed enums for controlled catalog values, validate Template Versions with a reusable semantic-version schema, validate updated dates as ISO calendar dates, and reject unknown keys. Registry generation and CI will fail when either the standard shadcn Registry Record or FormMuse Metadata is invalid.

Publication Status will use exactly `draft`, `published`, and `deprecated`. Draft templates are available only in development and are excluded from the production catalog and public installation output. Published templates are visible, searchable, documented, and installable only after passing every required quality check. Deprecated templates retain their stable Template Page, changelog, warning, and optional replacement reference, but are excluded from ordinary catalog results and featured sections. Direct installation remains available only while the deprecated source is safe, legally redistributable, and functional; a critical security, licensing, or compatibility issue disables installation while preserving the historical documentation page.

Every Form Template will declare exactly one Primary Category as the sole value in the standard top-level `categories` array. V1 categories are `contact`, `newsletter`, `waitlist`, `quote-request`, `project-inquiry`, `booking`, `consultation`, and `feedback`; the template's dominant visitor intent determines the choice. Secondary use cases, audiences, styles, and behaviours belong in `meta.formmuse.tags`, and the Primary Category must not be duplicated as a tag.
