# Standardize form behavior and submission

Every Form Template will use React Hook Form 7.82.0, Zod 4.4.3, and `@hookform/resolvers` 5.4.0 as its form-behavior baseline. Each template will provide accessible validation and visible submitting, success, and failure Form States, and expose `onSubmit: (values) => Promise<void>` using its exported values type. A resolved promise indicates success; a thrown error or rejected promise indicates failure. No custom FormMuse response object is required.

FormMuse will own the visible submission lifecycle: it will prevent duplicate submissions while a request is pending, preserve entered values after failure, and display safe template-defined failure copy rather than exposing a raw thrown error to the visitor. Adopters connect any backend or provider behind `onSubmit`; FormMuse will not provide or select a backend. The shared dependencies add some weight to every template, but they give adopters and coding agents one predictable, typed integration contract across the Curated Library.

After a resolved submission, the designed success state will remain visible indefinitely. FormMuse will not automatically clear, redirect, dismiss, or return to the form. A Form Template may provide an explicit action such as “Send another” when repeated submission makes sense; that action resets both the form values and submission state. V1 will not add separate `onSuccess` or `onError` props because adopters can perform analytics, navigation, and other side effects inside `onSubmit` before it resolves.

Every Form Template will expose `defaultValues?: Partial<FormValues>` as Initial Values. Each template will define complete internal defaults, merge the supplied partial values once when the component mounts, and retain that merged snapshot for explicit reset actions such as “Send another.” The prop is not controlled and later changes will not reinitialize or alter the form. Dynamic controlled values and reactive default-value APIs are outside V1.

Validation Timing will be submit-first with change-based correction. React Hook Form will use `mode: "onSubmit"`, `reValidateMode: "onChange"`, and first-error focus behavior. Before the first submit attempt, the template will not show validation errors; that submit validates the complete form and focuses the first invalid control. Once errors are visible, editing an erroneous field revalidates it and removes corrected feedback promptly.

The `<form>` will use `noValidate` so Zod and FormMuse render consistent designed messages instead of browser-specific popups. Templates will still retain semantic input types, native `required` where applicable, `aria-required` for custom controls when needed, `aria-invalid`, and programmatic links between controls, descriptions, and errors. Custom shadcn controls must forward focus correctly so first-error focus works for keyboard and assistive-technology users.

While `onSubmit` is pending, the form will set `aria-busy="true"`, disable its controls through a native `<fieldset disabled>`, and show visible progress text such as “Sending…”; a decorative spinner may accompany but never replace text. React Hook Form captures the validated values before the awaited call, so disabling controls afterward does not alter the Submission Connection. A polite `role="status"` for pending progress will sit outside the busy form subtree so `aria-busy` cannot defer that announcement.

After failure, the fieldset will be re-enabled, values preserved, and focus moved once to a visible failure heading or summary with `tabIndex={-1}`. After success replaces the form, focus will move once to the designed success heading with `tabIndex={-1}`. Focus is the single result announcement method: focused result content will not also use `role="alert"` or another live region that repeats the same message. Semantic state changes happen immediately; animation follows state and never delays accessibility updates.
