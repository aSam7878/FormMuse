# Use shadcn-first form controls

FormMuse will use shadcn components as the default Control Primitives for standard form controls, including Input, Textarea, Label or Field, Button, Checkbox, Radio Group, Select, Calendar, Popover, and Command. Base UI will be the sole officially supported Component Foundation in V1. Form Templates will import controls through adopter-owned local shadcn component paths such as `@/components/ui/input` rather than importing directly from `@base-ui/react`.

Each Form Template will declare and install only the controls and dependencies it uses, and template-specific visual styling will be scoped so it does not globally redefine an adopter's controls. Semantic HTML remains appropriate for the form element, structural and decorative markup, and cases where shadcn has no suitable component. FormMuse will not install an entire design system, introduce giant visual configuration objects, or automatically overwrite an adopter's existing shadcn files.

Template documentation, compatibility metadata, and Agent Handoffs will state the Base UI requirement explicitly. FormMuse will not create, guarantee, document, or test separate Radix variants in V1. Radix support may be considered later if genuine adopter demand justifies the additional authoring, registry, documentation, and quality-gate complexity. This policy gives templates a predictable, accessible, agent-friendly structure while preserving adopter ownership and keeping installation proportional to each template.

When shadcn is missing, documentation will initialize it explicitly with `shadcn@latest init --base base` before running the separate FormMuse item installation command. Existing Radix projects receive an unsupported warning; FormMuse will never automatically migrate or reconfigure their component foundation. Coding agents and troubleshooting guides should inspect project configuration with `shadcn info`, but ordinary human installation does not require that diagnostic step.
