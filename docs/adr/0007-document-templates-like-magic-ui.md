# Document Form Templates like Magic UI

Every FormMuse Template Page will follow the Magic UI component-documentation model: primary Preview and Code tabs; CLI and Manual Installation with pnpm, npm, Yarn, and Bun commands; additional examples; final usage code; and a documented Props table. Every Form Template will expose useful shared props such as `onSubmit`, `className`, and `defaultValues`, plus a limited set of template-specific props where they improve real reuse. FormMuse will avoid giant configuration objects and props for every visual detail because adopters own and can directly edit the installed source.

V1 will use one shared, statically generated Template Page layout rather than maintaining a separate MDX file for every Form Template. The page will be populated from its Registry Record, Distributed Template Source, Template Preview route, and referenced type-checked example files. Template-specific documentation under `meta.formmuse` will remain concise and structured—for example props, examples, usage notes, and accessibility notes—rather than storing long prose in JSON. Longer cross-template guides such as connecting a backend, customizing templates, and using FormMuse with coding agents will remain ordinary documentation pages.

Every Template Page will place a compact Installation Preflight before its CLI and Manual Installation tabs. It will state the React, TypeScript, Tailwind CSS 4, and shadcn Base UI requirements; provide pnpm, npm, Yarn, and Bun `shadcn@latest init --base base` commands when shadcn is not initialized; and clearly mark existing Radix projects as unsupported in V1 without migrating or reconfiguring them. `shadcn info` will be recommended for coding agents and troubleshooting, not required for every human installation.

Templates will ship with complete designed default copy and may expose a small number of optional template-specific Content Slots only where the design has natural reusable text, such as a title, description, or success message. FormMuse will not require identical copy props across templates, create a universal `content` object, use render props for ordinary wording, or expose every label, hint, button, and decorative phrase. Validation and safe failure copy remain template-authored. Every public Content Slot must appear in `meta.formmuse` and the generated Props table; deeper copy customization and localization remain adopter source edits in V1.
