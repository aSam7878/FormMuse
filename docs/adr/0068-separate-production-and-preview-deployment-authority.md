# Separate production and preview deployment authority

GitHub Actions will be the sole production build and verification authority for FormMuse. After every required check passes on `main`, the production workflow will deploy the exact verified static `out/` artifact to the Initial Production Host, Hostinger Premium. The production job will use a protected GitHub environment, minimum workflow permissions, one-at-a-time deployment concurrency, and a dedicated Production Deployment Credential with the narrowest access the actual Hostinger plan technically permits.

FormMuse will not claim in advance that Hostinger Premium provides an independently directory-scoped SFTP user. Current Hostinger documentation associates SFTP with the main FTP/SSH identity while separately documenting directory-scoped additional FTP accounts. The Hostinger capability audit must determine whether an SSH key, SSH-only password, SFTP identity, or another supported credential offers the best practical isolation. Only the production deployment job may receive that credential, and the repository, preview system, untrusted pull requests, and build output must never contain it.

Cloudflare Pages will be the Preview Host only. Its Git integration may independently rebuild trusted same-repository branches and pull requests, but automatic Cloudflare production-branch deployment will be disabled. Preview builds receive only preview-safe configuration, remain `noindex`, and receive no production credential. Pull requests from forks receive the normal non-secret CI suite and no credential-bearing preview deployment.

Cloudflare preview builds and GitHub production builds must use the same pinned Node.js and pnpm versions, committed lockfile, frozen installation, build command, and validated environment contract. A passing Cloudflare build is preview evidence, not the Production Artifact; GitHub Actions remains authoritative for production.

The Hostinger deployment prototype must prove partial-deployment protection, uploaded-file verification, serial production deployment, and rollback to at least the previous successful Production Artifact. Prefer uploading into a temporary or versioned release location and promoting only a complete verified release. The exact safe promotion and rollback commands depend on capabilities observed in the real Hostinger account and will be recorded during the Hanging Gifts vertical slice rather than assumed in architecture documentation.
